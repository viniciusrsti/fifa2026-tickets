**eu quero** completar a Fase 5 em 8h expondo 3 tools MCP (`consultar_disponibilidade`, `verificar_ingresso`, `consultar_bracket`) via Function .NET e integrando um componente React de chatbot no frontend conversando com Gemini 2.0 Flash,

## Dev Notes

### Arquitetura técnica delta

```
[Browser — frontend Vite]
   │ usuário digita "Tem ingresso pra Brasil x Argentina?"
   ▼
[Chatbot.tsx (React)]
   │ useLlmChat('gemini')
   ▼
[Gemini 2.0 Flash API]
   │ recebe pergunta + tools list (MCP schema)
   │ decide chamar consultar_disponibilidade
   ▼
[Gateway YARP — Container App por aluno]          ← OBRIGATÓRIO (herança de F2/F3)
   │ valida Bearer token (AddJwtBearer)
   │ propaga X-Entra-OID
   ▼
[MCP Server Function: POST /mcp JSON-RPC]
   │ tools/call → consultar_disponibilidade(matchId)
   │ lê X-Entra-OID para logging (não revalida JWT)
   │ Dapper SELECT em SQL (purchases, matches)
   ▼
[Gemini]
   │ recebe resultado da tool
   │ gera resposta natural
   ▼
[Chatbot UI]
   │ exibe resposta ao usuário
```

**Nota importante:** o MCP Server é uma Function .NET acessível VIA o gateway YARP — não direto do browser. Isso mantém o gateway como ponto único de autenticação/autorização (ADE-004/ADE-005 Inv 4) e é o caminho pedagógico correto ("tudo passa pelo gateway"). O chatbot envia o Bearer token do usuário logado (obtido via MSAL.js em F3).

### MCP tools schema (JSON Schema — rastreado a modelcontextprotocol.io)

```json
{
  "name": "consultar_disponibilidade",
  "description": "Consulta disponibilidade e preços de ingressos para uma partida da Copa 2026",
  "inputSchema": {
    "type": "object",
    "properties": {
      "matchId": { "type": "integer", "description": "ID numérico da partida" },
      "matchDescription": { "type": "string", "description": "ex: Brasil x Argentina" }
    },
    "required": []
  }
}
```

Pattern análogo para `verificar_ingresso` (recebe `ingressoId`) e `consultar_bracket` (recebe `rodada`).

### Conteúdo bônus: portabilidade entre LLMs

SPEAKER-NOTES inclui demo de 5min: aluno troca `LLM_PROVIDER=gemini` → `groq` → `mistral` e a mesma pergunta produz respostas equivalentes via o mesmo MCP Server. Demonstra que o protocolo MCP desacopla o LLM dos dados.

### Troubleshooting conhecido

| Sintoma | Causa provável | Solução |
|---|---|---|
| Gemini não encontra as tools | `tools/list` retorna formato inválido | Validar schema JSON-RPC contra spec MCP; conferir Content-Type: application/json |
| 401 ao chamar MCP Server | Bearer token não enviado ou expirado | Verificar MSAL.js renova token antes de expirar; relogar se necessário |
| X-Entra-OID ausente no MCP Server | Gateway não propagou (token não autenticado) | Confirmar usuário está logado; gateway propaga OID apenas para requests autenticados |
| Timeout de LLM (> 10s) | Groq/Gemini latência ou quota | Usar fallback conforme AC-12; mostrar loading state no UI |
| Tool call não dispara | LLM não reconheceu intenção de consulta | Ajustar description da tool (mais contexto sobre quando usar) |
