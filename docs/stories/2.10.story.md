## Story

**Como** aluno/instrutor do workshop "Living Lab Azure-Native",
**eu quero** que o fluxo de compra assíncrona v2 (Azure Service Bus + Functions) processe o carrinho inteiro (N partidas/categorias/quantidades em um único POST) em vez de aceitar apenas 1 ingresso por vez,
**para que** o sistema reflita um cenário real de e-commerce de ingressos — onde o comprador monta um carrinho com múltiplos itens antes de confirmar — e o workshop demonstre o padrão de fan-out via Service Bus com correlação de pedido (orderId).

## Acceptance Criteria

1. **AC-1 — Endpoint aceita carrinho com N itens:** POST `/api/v2/purchase` aceita o shape `{ userId, items: [{matchId, category, quantity}, ...] }` com 1 a 20 linhas (limite por compra definido em `PurchaseRequest.MaxItems = 20`). Cada linha gera uma mensagem distinta no Service Bus `tickets-purchase` (fan-out via output binding `string[]`). N linhas → N compras gravadas no SQL (tabela `purchases`), cada uma com seu `correlationId` distinto mas o mesmo `orderId`.

2. **AC-2 — Backward-compat: shape legado single ainda funciona:** O endpoint aceita TAMBÉM o shape legado `{ matchId, category, userId, quantity }` (sem `items`). O método `PurchaseRequest.Normalize()` converte o shape legado para `items` com 1 linha antes da validação. O smoke test do workflow n8n (que envia o shape legado) NÃO quebra.

3. **AC-3 — Response 202 com orderId e correlationIds:** A resposta de sucesso é `{ orderId, status: "queued", correlationIds: [guid, ...], correlationId? }`. O campo `orderId` (Guid) agrupa todas as mensagens do mesmo carrinho. O campo `correlationIds` (array) lista todos os IDs gerados (1 por linha). O campo `correlationId` (Guid, singular) está presente APENAS quando `items.length == 1` (iguala `correlationIds[0]`) para backward-compatibility com o smoke test do workflow n8n (`jq -e '.correlationId'`).

4. **AC-4 — All-or-nothing na validação:** Todos os itens do carrinho são validados ANTES de qualquer mensagem ser publicada. Se qualquer item falhar na validação (DataAnnotations), NENHUMA mensagem é publicada e o endpoint retorna `400` com detalhes de qual item falhou (`items[i]: <erro>`). A semântica é tudo-ou-nada — não existe publicação parcial.

5. **AC-5 — X-Entra-OID propagado a TODAS as mensagens do carrinho:** O header `X-Entra-OID` é lido uma única vez pelo Entry e replicado em `PurchaseMessage.EntraOid` de CADA mensagem publicada. O campo `EntraOid` em `PurchaseMessage` é `Guid?` (mantém o tipo existente — é `Guid?` e não `string?` diferente do McpServer). Ausente/inválido → `null` em todas as mensagens (fluxo sem identidade Entra — backward-compat S2.3).

6. **AC-6 — PurchaseConsumerFunction NÃO muda:** O consumer já processa mensagens per-mensagem e a constraint UNIQUE em `correlationId` (Story 2.7) já garante idempotência. Nenhuma linha de `PurchaseConsumerFunction.cs` é modificada. `PurchaseStatusFunction.cs` também não muda.

7. **AC-7 — PurchaseItem: validação quantity 1-10 por linha:** Cada item do carrinho valida `quantity` entre 1 e 10 (por linha — não é o total do carrinho). `category` aceita apenas `VIP | Cat1 | Cat2` (regex, validado por `DataAnnotations`). `matchId` deve ser positivo (>= 1).

8. **AC-8 — Frontend: tipos atualizados para suportar N correlationIds:** `apiFunctionV2.ts` define os tipos novos: `PurchaseCartItem` (1 linha do carrinho), `PurchaseCartRequest` (envelope: `{ userId, items: PurchaseCartItem[] }`), `PurchaseFunctionV2Accepted` atualizado com `orderId: string`, `correlationIds: string[]`, `correlationId?: string`. `Checkout.tsx` (`handleSubmitV2`) envia o carrinho inteiro (todos os `items`, não apenas `items[0]`). O polling de status existe para CADA `correlationId` do array (ou exibe um resumo por `orderId`).

9. **AC-9 — Sem migration SQL:** Nenhuma migration é necessária. O schema existente já suporta o padrão: `purchases` recebe N linhas com `correlationId` distintos e mesmo valor de referência de pedido. O campo `order_id` (se existir no schema) pode ser usado; se não existir, `orderId` é apenas rastreabilidade na mensagem/log (não exige coluna nova). Confirmar por inspeção do schema — NÃO adicionar coluna se ela não existir.

10. **AC-10 — v1 (api.ts síncrono) intacto:** O fluxo de compra v1 (`POST /api/purchase` via Node/Express, `src/lib/api.ts`) não é modificado. O toggle de fluxo (presença de `VITE_FUNCTION_V2_URL`) continua funcionando. Nenhum arquivo de v1 é tocado.

11. **AC-11 — Testes multi-item no Functions.Tests:** Os testes de `PurchaseEntryFunctionTests.cs` são atualizados para o novo contrato da `EntryOutput` (que agora usa `Messages` array em vez de `Message` singular). Novos testes cobrem: (a) carrinho com 3 itens → `Messages.Length == 3`, todos com mesmo `orderId`, correlationIds distintos; (b) all-or-nothing: 1 item inválido no carrinho de 3 → 400 e `Messages == null`; (c) backward-compat: shape legado single → `Messages.Length == 1`, `correlationId` singular presente na response; (d) X-Entra-OID replicado em TODAS as mensagens do carrinho. Contagem alvo: Functions.Tests de 50 → >= 58 testes (8+ novos). `dotnet test` verde sem regressões em Gateway (11/11) e McpServer (70+).

12. **AC-12 — PII / segurança:** `EntraOid` (Guid? da mensagem) nunca é logado em texto — apenas `hasEntraIdentity = entraOid.HasValue` (booleano). O `orderId` é logado via `BeginScope` (App Insights agrupa). Os `correlationId` de cada linha são logados individualmente. Sem hardcode de connection strings (usa `ServiceBusConnection` via configuração).

## Fora do escopo desta story

- **Tabela de pedidos:** Não será criada nenhuma tabela `orders` ou coluna `order_id` no SQL. O `orderId` é apenas rastreabilidade na camada de mensageria/log (ADE-000 Inv 5). Se o schema já tiver `order_id`, o consumer pode preenchê-lo; mas nenhuma migration é autorizada por esta story.
- **Limite de carrinho configurável:** O máximo de 20 itens (`MaxItems`) é constante no código — não é App Setting. Não implementar configuração dinâmica.
- **Rollup de status por orderId:** O `PurchaseStatusFunction` existente consulta por `correlationId` individual. Não implementar endpoint `/api/v2/order/{orderId}` com rollup (story futura).
- **Retry automático no frontend:** O polling de status por `correlationIds` pode ser simplificado (exibir todos os IDs, não implementar retry automático de cada um).
- **v1 e gateway YARP:** Nenhuma alteração no fluxo de compra v1 nem no gateway YARP/McpServer.

## Dev Notes

### Decisão arquitetural: Opção B — fan-out no Entry (não no Consumer)

**Rejeitada:** Opção A (frontend faz N chamadas paralelas) — aumenta acoplamento e torna o carrinho não-atômico do ponto de vista do cliente.

**Adotada:** Opção B — 1 POST atômico do frontend com o carrinho inteiro → `PurchaseEntryFunction` gera `orderId` (1) + `correlationId` (N, um por linha) → publica N mensagens via `string[]` no output binding → `PurchaseConsumerFunction` processa cada mensagem independentemente (sem mudança).

**Por que `string[]` no output binding?** O Azure Functions SDK (worker isolado) com `[ServiceBusOutput]` em `string[]` publica CADA elemento como mensagem individual no Service Bus. Isso é o mecanismo "fan-out declarativo" — sem `ServiceBusSender` imperativo, sem Azure SDK adicional.

**Por que o Consumer NÃO muda?** O consumer já é per-mensagem e a constraint `UNIQUE(correlationId)` já garante idempotência. Adicionar `OrderId` ao `PurchaseMessage` (nullable) não quebra a desserialização existente — o consumer simplesmente ignora o campo desconhecido (ou grava se o schema tiver a coluna, sem exigir migration).

**Gotcha crítico — output binding `string[]` vs `string?`:**
A story 2.2 / 2.3 usava `string? Message` (singular) no `EntryOutput`. A mudança para `string[]? Messages` (array) exige atualizar TODOS os testes que referenciam `output.Message`. Se o teste tentar `output.Message` (propriedade que não existe mais), compilará com erro. Isso é um breaking change intencional e esperado — os testes precisam ser atualizados como parte desta story (AC-11, Task 6).

**Gotcha crítico — DataAnnotations não desce em coleções:**
`[MinLength]`/`[MaxLength]` em `List<PurchaseItem>` valida o tamanho da coleção, mas `Validator.TryValidateObject` não valida recursivamente cada `PurchaseItem`. Por isso o `PurchaseEntryFunction` itera explicitamente os itens e valida cada um com um `ValidationContext` separado (Task 4.3).

**Gotcha — all-or-nothing e output binding:**
O output binding só publica quando `Messages` é não-nulo e não-vazio. Em qualquer caminho de erro 400, `Messages` deve permanecer `null` (valor default do campo). Nunca publicar uma mensagem parcialmente válida.

**Schema SQL — campo `order_id`:**
Verificar se `purchases` tem coluna `order_id`. Se sim: o consumer pode preencher com `message.OrderId` sem migration. Se não: o `OrderId` é rastreabilidade apenas na mensagem/log — nenhuma coluna nova é criada. Esta story NÃO autoriza migration.

### Invariantes herdadas obrigatórias

| Invariante | Origem | Obrigação para esta story |
|---|---|---|
| `[HttpResult]` obrigatório em multi-output | Fix Story 2.7 (commit com correção de `[HttpResult]`) | Manter o atributo `[HttpResult]` no `HttpResponse`. Sem ele, a response HTTP é ignorada pelo host (bug silencioso). |
| `ServiceBusConnection` via configuração | ADE-000 (sem hardcode de connection strings) | O binding usa `Connection = "ServiceBusConnection"` — lido de App Settings. Nunca hardcoded. |

| Consumer inalterado (UNIQUE correlationId já existe) | S2.2 / S2.7 | Não modificar `PurchaseConsumerFunction.cs`. A idempotência por `correlationId` já está no banco. |

### Arquivo de referência para o output binding array

O padrão de `string[]` no output binding foi validado empiricamente no projeto (a implementação já está em andamento). A estrutura `EntryOutput` com `[ServiceBusOutput]` em `string[]?` é a mecânica correta para o fan-out declarativo no worker isolado do Azure Functions .NET 8.
