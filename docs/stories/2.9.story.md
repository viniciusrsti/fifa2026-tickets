## Story

**Como** aluno/instrutor do workshop "Living Lab Azure-Native",
**eu quero** adicionar a primeira tool MCP de ação (`criar_alerta_ingresso`, `ReadOnly=false`) ao chatbot e configurar um workflow agêntico no n8n (AI Agent node + MCP Client Tool) que, ao receber o alerta via webhook, consulta disponibilidade via MCP e redige a notificação de forma autônoma,
**para que** o chat demonstre o padrão de "dois agentes cooperando" — o front (Gemini + 8 tools) decide *o quê* fazer; o AI Agent do n8n decide *o como* executar — materializando didaticamente a separação entre sentidos (leitura) e mãos (ação), e provando que ações agênticas não violam a regra de ouro (nenhuma escrita direta no SQL).

   - **V-4:** FQDN/porta interna do McpServer acessível pelo n8n dentro do mesmo Container Apps environment (DNS interno — topologia leste-oeste, Invariante 7, bypass do gateway YARP).
   Os achados de V-1..V-4 devem ser registrados **antes** de qualquer codificação e usados como contrato de configuração do workflow (ACs 7/8 dependem deles).

2. **AC-2 — Tool `criar_alerta_ingresso` implementada (ADE-006 B.1, Mão):** método estático adicionado à classe `FifaTicketTools` (arquivo real `Tools/FifaTickerTools.cs`, ou nova classe `FifaActionTools.cs` na mesma pasta — a critério do @dev) com:
   - `[McpServerTool(Name = "criar_alerta_ingresso")]` **sem** `ReadOnly = true` (ReadOnly omitido ou `ReadOnly = false` — esta é a primeira "mão"; padrão ReadOnly=false é o default do SDK conforme ADE-006 Inv 2).
   - `[Description]` em PT-BR: `"Cria um alerta para avisar quando ingressos ficarem disponíveis para uma partida. Aciona uma automação de orquestração no n8n. Use quando o usuário pedir para ser avisado ou monitorar disponibilidade de ingresso para um jogo."` (texto exato).
   - Parâmetros: `matchId` (int?, opcional, `= null`), `matchDescription` (string?, opcional, `= null`), `categoria` (string?, opcional, `= null` — rótulo real via `CategoryLabelMapper.ToDbLabel()` se preenchido). Pelo menos um de `matchId`/`matchDescription` deve ter valor; a tool valida isso no corpo (retorna erro descritivo se ambos null).
   - Outros parâmetros DI (padrão): `IAlertWebhookNotifier alertNotifier`, `EntraOidContext oidContext`, `ILogger<DiagnosticsCategory> logger`, `CancellationToken cancellationToken = default`.
   - Corpo: log mascarado (`oid` + parâmetros de intent), chama `alertNotifier.NotifyAlertAsync(payload, ct)`, retorna `AlertResult { Registrado = true, Mensagem = "Alerta registrado. Você será notificado quando houver disponibilidade." }` em sucesso; `AlertResult { Registrado = false, Mensagem = "..." }` em falha do webhook (não re-throw — fire-and-forget, ADE-006 B.1).

3. **AC-3 — Payload do webhook de alerta (ADE-006 B.1/Invariante 4):** o corpo JSON enviado pela tool ao endpoint n8n deve incluir exatamente: `{ correlationId, entraOid, matchId, matchDescription, categoria, requestedAt }` onde:
   - `correlationId`: GUID novo gerado na tool (rastreabilidade futura para F6 — ADE-000 Inv 5).
   - `entraOid`: lido de `EntraOidContext.GetRawOid()` (retorna `string?` — método público **já existente** no código real; ver Gotcha 2). Propaga Invariante 5/6/7 — identidade como dado, não credencial; nunca logado em texto. Tipo no DTO: `string?` (o header `X-Entra-OID` é uma string; não há parsing para `Guid` no McpServer).
   - `matchId`: int? passado pelo LLM (null se não fornecido).
   - `matchDescription`: string? passado pelo LLM (null se não fornecido).
   - `categoria`: rótulo real (`'VIP Premium'`/`'Categoria 1'`/`'Categoria 2'`) via `CategoryLabelMapper.ToDbLabel(categoria)` — se `categoria` não mapeado ou null, enviar null (não inventar rótulo).
   - `requestedAt`: timestamp UTC ISO 8601 (`DateTimeOffset.UtcNow`).
   A URL do webhook é lida de App Setting `N8N_ALERT_WEBHOOK_URL` (nome distinto de `N8N_WEBHOOK_URL` de F4 para evitar conflito de workflows no n8n). Se ausente/vazia: no-op silencioso, retorna `AlertResult { Registrado = false, Mensagem = "Webhook de alerta não configurado neste ambiente." }` (graceful degradation — padrão de `N8nWebhookNotifier` de F4).

4. **AC-4 — `IAlertWebhookNotifier` + implementação `AlertWebhookNotifier` no McpServer:** nova interface `IAlertWebhookNotifier` em `src/Fifa2026.V2.McpServer/Data/` com método `Task<bool> NotifyAlertAsync(AlertWebhookPayload payload, CancellationToken ct)`. Implementação `AlertWebhookNotifier`: usa `IHttpClientFactory` (named client `"n8n-alert"`, timeout 5s, `CancelAfter` encadeado ao CancellationToken do host), `PostAsJsonAsync` na URL do App Setting `N8N_ALERT_WEBHOOK_URL`. Qualquer falha (timeout, rede, non-2xx) retorna `false` — **nunca re-throw** (fire-and-forget). Registrado em `Program.cs` via `builder.Services.AddHttpClient("n8n-alert", c => c.Timeout = TimeSpan.FromSeconds(5))` + `builder.Services.AddScoped<IAlertWebhookNotifier, AlertWebhookNotifier>()`.

5. **AC-5 — `AlertWebhookPayload` DTO e `AlertResult` DTO:** `AlertWebhookPayload` em `src/Fifa2026.V2.McpServer/Data/McpQueryResults.cs` (ou arquivo próprio `AlertPayloads.cs`) com `[JsonPropertyName]` PT-BR para todos os campos de AC-3. `AlertResult` com `[JsonPropertyName]`: `registrado` (bool), `mensagem` (string). Ambos com inicialização obrigatória (record ou init-only props) para evitar warnings de nullable.

6. **AC-6 — `tools/list` retorna 8 tools após a implementação:** chamada `tools/list` lista exatamente 8 tools: as 7 read-only existentes (`consultar_disponibilidade`, `verificar_ingresso`, `consultar_bracket`, `consultar_partidas`, `consultar_classificacao`, `consultar_time`, `consultar_estadio`) + `criar_alerta_ingresso`. O discriminador formal: 7 com `readOnly: true` no schema retornado pelo SDK + 1 com `readOnly: false` (ou ausente) — verificável por inspeção do JSON retornado pelo `tools/list`.

7. **AC-7 — Workflow n8n com AI Agent node + MCP Client Tool (ADE-006 B.2) — configurado com nomes/transporte verificados em AC-1:** usando os achados de V-1..V-4 (AC-1), @devops configura no n8n UI um workflow `chat-alert-ingresso` com:
   - **Webhook trigger** recebendo `{ correlationId, entraOid, matchId, matchDescription, categoria, requestedAt }` no body (AC-3).
   - **Nó AI Agent** (nome exato verificado em V-1) com um LLM (Gemini — credencial configurada conforme V-3) como cérebro.
   - **MCP Client Tool** (nome/transporte verificados em V-2) apontando para o McpServer interno via FQDN/porta verificados em V-4 (bypass do gateway — Invariante 7 leste-oeste).
   - O AI Agent usa o MCP Client Tool para consultar `consultar_disponibilidade` (e opcionalmente `consultar_partidas`) com os dados do payload — sentidos read-only (regra de ouro no Agente 2).
   - **HTTP Request node** (ou equivalente) como mock de notificação — `httpbin.org/post` com payload `{ correlationId, entraOid, mensagem_redigida }` propagando `correlationId` (rastreabilidade F6).
   - `correlationId` propagado em **todos os nodes de log** do workflow (padrão de F4 AC-7).

8. **AC-8 — Regra de ouro preservada no Agente 2 (ADE-006 Inv 1/4):** as tools disponíveis para o AI Agent do n8n via MCP Client Tool são os **mesmos sentidos read-only** — `ReadOnly=true`. Verificável por inspeção da configuração do MCP Client Tool no n8n (lista de tools visível na interface). Nenhuma escrita SQL ocorre dentro do workflow n8n.

### Padrão de HttpClient a seguir (N8nWebhookNotifier de F4)

O padrão de referência para `AlertWebhookNotifier` está em `src/Fifa2026.V2.Functions/Data/N8nWebhookNotifier.cs`. Replicar a mesma estrutura no McpServer (assemblies independentes, sem projeto compartilhado — padrão igual ao `CategoryLabelMapper`):

```csharp
// Estrutura base de AlertWebhookNotifier (mirror de N8nWebhookNotifier)
public sealed class AlertWebhookNotifier : IAlertWebhookNotifier
{
    public const string WebhookUrlSetting = "N8N_ALERT_WEBHOOK_URL";  // distinto de N8N_WEBHOOK_URL
    private static readonly TimeSpan WebhookTimeout = TimeSpan.FromSeconds(5);

    private readonly IHttpClientFactory _httpClientFactory;
    private readonly string? _webhookUrl;
    private readonly ILogger<AlertWebhookNotifier> _logger;

    public AlertWebhookNotifier(IHttpClientFactory httpClientFactory,
        IConfiguration configuration, ILogger<AlertWebhookNotifier> logger)
    {
        _httpClientFactory = httpClientFactory;
        _webhookUrl = configuration[WebhookUrlSetting];
        _logger = logger;
    }

    public async Task<bool> NotifyAlertAsync(AlertWebhookPayload payload, CancellationToken ct = default)
    {
        if (string.IsNullOrWhiteSpace(_webhookUrl))
        {
            _logger.LogDebug("{Setting} não configurado — disparo de alerta ignorado.", WebhookUrlSetting);
            return false;
        }
        try
        {
            using var client = _httpClientFactory.CreateClient("n8n-alert");
            using var cts = CancellationTokenSource.CreateLinkedTokenSource(ct);
            cts.CancelAfter(WebhookTimeout);
            using var response = await client.PostAsJsonAsync(_webhookUrl, payload, cts.Token);
            if (response.IsSuccessStatusCode) { /* log success */ return true; }
            /* log warning non-2xx */ return false;
        }
        catch (Exception ex)
        {
            _logger.LogWarning(ex, "Falha ao disparar webhook de alerta.");
            return false;  // NUNCA re-throw — fire-and-forget
        }
    }
}
```

### Gotchas críticos de implementação

**Gotcha 1 — `N8N_ALERT_WEBHOOK_URL` é DIFERENTE de `N8N_WEBHOOK_URL`:**
F4 usa `N8N_WEBHOOK_URL` para o workflow `post-purchase-notification`. A nova tool deve usar `N8N_ALERT_WEBHOOK_URL` para apontar ao workflow `chat-alert-ingresso` — endpoints de webhook distintos no n8n. Misturar os dois faria a compra acionar o alerta e vice-versa. Verificar na `AlertWebhookNotifier` que a constante `WebhookUrlSetting = "N8N_ALERT_WEBHOOK_URL"` (com `_ALERT_`).

**Gotcha 2 — `EntraOidContext.GetRawOid()` JÁ existe (verificado pelo @po; NÃO criar método novo):**
`EntraOidContext` (`src/Fifa2026.V2.McpServer/Tools/EntraOidContext.cs`) expõe **dois** accessors públicos: `GetMaskedOidForLog(): string` (mascarado, para log — usado pelas 7 tools atuais) **e `GetRawOid(): string?`** (valor bruto do header, linha 30). Para o payload do webhook use **`GetRawOid()`** — **NÃO** adicione `GetOidOrNull()` (não existe e não é necessário). Tipo de retorno é `string?` → `AlertWebhookPayload.EntraOid` é `string?` (NÃO `Guid?`, diferente do `N8nWebhookPayload` de F4 que faz parse para Guid; o McpServer não faz parse). O uso do OID raw no payload é **autorizado pelo ADE-006 Inv 7 v1.1** ("entraOid viaja como dado no payload") — é rastreável ao ADE, não invenção (Art. IV). **Invariante de segurança:** `GetRawOid()` alimenta SOMENTE o payload; todo log permanece com `GetMaskedOidForLog()` — o valor raw NUNCA entra em log (Task 9.4 verifica isso).

**Gotcha 3 — Build ACR requer limpar `obj/bin` (herdado de S2.8):**
`Remove-Item -Recurse -Force src/Fifa2026.V2.McpServer/obj, src/Fifa2026.V2.McpServer/bin -ErrorAction SilentlyContinue` antes do `az acr build`.

**Gotcha 4 — n8n `latest` = verificar workflow no início de cada aula:**
A imagem `n8nio/n8n:latest` foi escolhida pelo owner (decisão S2.4 AC-3, commit f888ab5) para sempre rodar a versão mais nova. Trade-off aceito: o AI Agent node pode mudar de UI/nome em versões futuras. Mitigação: validar o workflow `chat-alert-ingresso` no início de cada aula (padrão estabelecido em S2.4).

**Gotcha 5 — gateway cache é GET-only (fix e5451ce):**
POST /mcp não é cacheado. `criar_alerta_ingresso` é um `tools/call` (POST) — sem conflito. Não é necessária nenhuma ação.

**Gotcha 6 — AI Agent do n8n enxerga todas as 8 tools do MCP Client Tool, incluindo `criar_alerta_ingresso`:**
Esta é a situação esperada pela topologia: o MCP Client Tool conecta ao McpServer interno que lista 8 tools. Para garantir a regra de ouro (AC-8), o @devops DEVE verificar no n8n UI que o AI Agent **não invoca** `criar_alerta_ingresso` durante o workflow `chat-alert-ingresso` — a tool de ação não deve ser chamada recursivamente pelo agente. Uma alternativa mais segura é limitar as tools do AI Agent via configuração do MCP Client Tool (se o n8n permitir seleção de subset). Registrar a decisão no AC-7 após verificação.

**Gotcha 7 — `correlationId` gerado na tool vs propagado do gateway:**
O gateway YARP injeta `X-Correlation-ID` como header (ADE-004). A tool `criar_alerta_ingresso` gera um **novo** GUID como `correlationId` para o alerta (contexto de rastreabilidade diferente do `X-Correlation-ID` da request HTTP). Em F6 (Fase C), pode ser interessante repassar o `X-Correlation-ID` original, mas isso exige que `EntraOidContext` (ou contexto equivalente) exponha o correlation ID do header. Para esta story, gerar um novo GUID é suficiente e mais simples — documentar como decisão (AUTO-DECISION).
