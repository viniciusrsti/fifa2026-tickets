## Dev Notes

### Princípio editorial-rector

> O workshop tem **um único autor narrativo** (Atlas/@analyst). Mesmo que 4 agentes produzam conteúdo, a voz deve ser indistinguível entre fases — como se um humano só tivesse escrito tudo.

### Diretrizes de tom (padrão estabelecido em F1-F3)

- 2ª pessoa singular: "você cria", "você verá", "anote a URL"
- Direto: evitar frases longas com 3+ subordinadas
- Pedagógico firme: "Por quê?" antes de "Como?" em cada conceito-chave
- Emoji parcimônio: callouts apenas, não decorativo

### Padrão de callouts (sincronizado com EPIC-001 / Story 1.5)

- 📋 (anote) — info que aluno precisa salvar (URL, key, IP)
- ✅ "Pronto quando:" — validação ao fim de cada Step
- 💰 (custo) — valor real, não estimativa vaga
- 🔐 (segredo) — onde armazenar (App Setting, Key Vault)
- ⚠️ (atenção) — armadilha conhecida
- 💡 (explicação) — "por que" pedagógico

### Nomenclatura editorial do re-escopo (OBRIGATÓRIA)

| Componente | Como escrever no material didático | NUNCA escrever |
|---|---|---|
| Gateway F2 | "Gateway YARP" ou "Gateway em código (.NET + YARP)" | "APIM", "Azure API Management" como componente do fluxo v2 |
| Nota cultural F2 | "Em produção corporativa, o equivalente gerenciado é o Azure API Management — aqui ensinamos o conceito em código para transparência" | Omitir essa nota (aluno merece saber que APIM existe) |
| Identidade F3 | "App Registration (tenant workforce do aluno) + MSAL.js" | "External ID", "CIAM tenant", "tenant externo" |
| Nota cultural F3 | "Em produtos B2C reais, o equivalente seria o Entra External ID — aqui usamos o tenant workforce para reduzir atrito" | Omitir essa nota |
| Claim de identidade | "`oid` (Object ID do Entra)" e "`entra_oid` (coluna)" | "GUID↔int mapping", "ADE-001" |
