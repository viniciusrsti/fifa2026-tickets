## Story

**Como** aluno do workshop "Living Lab Azure-Native",
**eu quero** completar a Fase 3 em 6h criando uma App Registration no meu próprio tenant Entra ID workforce, integrando MSAL.js no frontend SPA para login OIDC (Authorization Code + PKCE), e ativando a validação de JWT no gateway YARP (F2) com `AddJwtBearer` — preservando o JWT+bcrypt do v1 para comparação didática,
**para que** eu compreenda OIDC moderno (Authorization Code Flow + PKCE, claims, scopes, App Roles), a diferença entre autenticação local (v1: bcrypt+JWT HS256) e federada (v2: Entra OIDC RS256), e tenha o fluxo de identidade do v2 completo — com o gateway YARP como guardião único de JWT, base para F5 (chatbot) e F6 (Flow Visualizer).

## Dev Notes

### Arquitetura técnica delta

```
[Browser SPA — Vite/React]
   │ Login via MSAL.js (Authorization Code + PKCE)
   │ Obtém access_token (Entra tenant workforce)
   │
   │ POST /purchase + Authorization: Bearer <token>
   ▼
[Container App: gateway-<iniciais>  (ASP.NET Core + YARP)]
   ├─ AddJwtBearer("Entra"): valida iss/aud/assinatura
   ├─ Extrai claim oid → injeta header X-Entra-OID
   └─ MapReverseProxy → Function F1
            │
            ▼
     [Function PurchaseEntryFunction]
            ├─ Lê header X-Entra-OID
            ├─ INSERT purchases com entra_oid = <oid-do-token>
            └─ continua fluxo F1 (Service Bus → Consumer → SQL)
```

> **Gateway como guardião único:** a Function confia no header `X-Entra-OID` propagado pelo gateway — não valida o JWT novamente. Isso é intencional e documentado (ADE-005 Inv 4). O cliente nunca chama a Function diretamente (URL real não exposta — ADE-004 Inv 1/5).

### Requisitos de configuração de identidade (ADE-005 Invariante 5)

| Item | Detalhe |
|---|---|
| **App Registration** | Tipo: SPA (Authorization Code + PKCE, sem client_secret no browser) |
| **Platform** | Single-page application |
| **Redirect URI (dev)** | `http://localhost:5173` |
| **Redirect URI (prod)** | `https://fifa2026-web.azurewebsites.net` |
| **Scope exposto** | App ID URI → `purchase.write` (ou similar) |
| **Audience (`aud`)** | = `clientId` da App Registration (padrão MSAL SPA) |
| **App Roles** | `Admin`, `Operator`, `Viewer` (em App Reg admin separada) |
| **Social providers** | Google e/ou GitHub como identity providers federados |
| **Easy Auth (complementar)** | App Service Authentication → provider Microsoft → callback `/.auth/login/aad/callback`; secret gerenciado pelo App Service (ADE-005 Inv 2/5) |

### Schema delta (conforme ADE-005 Invariante 3 + ADE-000 Invariante 2)

```sql
-- phase-03.sql — idempotente
IF NOT EXISTS (SELECT 1 FROM sys.columns 
               WHERE Name='entra_oid' AND Object_ID=Object_ID('purchases'))
    ALTER TABLE purchases ADD entra_oid UNIQUEIDENTIFIER NULL;

IF NOT EXISTS (SELECT 1 FROM sys.indexes WHERE Name='IX_purchases_entra_oid')
    CREATE INDEX IX_purchases_entra_oid
        ON purchases(entra_oid)
        WHERE entra_oid IS NOT NULL;
```

> **Decisão de design registrada (AUTO-DECISION):** coluna em `purchases` (não `users`) — mantém comparação lado-a-lado v1/v2 na mesma tabela (ADE-000 Inv 1). Se @architect preferir `users.entra_oid`, ajuste no design final é simples e não invalida a story. Não há tabela de mapping `user_identity_mapping` (ADE-001 aposentada por ADE-005).

### Comparação v1 vs v2 (para README — AC-10, adaptada de ADE-005)

| Aspecto | v1 (intocado) | v2 (F3) |
|---|---|---|
| Storage de senha | bcrypt(10 rounds) em `users.password_hash` | Não armazenado (provider externo) |
| Token | JWT HS256 local (Express `jsonwebtoken`) | JWT RS256 do Entra workforce |
| Issuer | `fifa2026-api` local | `https://login.microsoftonline.com/<tenant-id>/v2.0` |
| Validação | Middleware Express manual | `AddJwtBearer` no gateway YARP |

| MFA | Não | Sim (configurável no tenant) |

### Troubleshooting esperado (mapa antecipado)

| Sintoma | Causa provável | Mitigação |
|---|---|---|
| Redirect URI incorreta (AADSTS50011) | URI cadastrada na App Registration não bate com `localhost:5173` ou URL de prod | Verificar Portal Entra ID → App Registration → Authentication → Redirect URIs; garantir `http://localhost:5173` (dev) e URL prod do App Service |
| `aud`/audience mismatch no token (401 com "Invalid audience") | `AddJwtBearer` configurado com `Audience` diferente do `clientId` da App Registration | Confirmar que `opt.Audience` = `clientId` da App Registration SPA (padrão MSAL SPA — o `aud` é o Client ID) |
| Token expirado (401 com "Lifetime validation failed") | Access token de 1h expirou; MSAL não renovou silenciosamente | Confirmar `acquireTokenSilent` com `forceRefresh: false` antes de cada chamada; token expirado é comportamento esperado (didático: cenário AC-12) |
| CORS bloqueando fluxo MSAL no browser | Redirect de login bloqueado por CORS do frontend ou do gateway | Garantir `AddCors` no gateway permite origem do frontend; MSAL.js usa popup/redirect — o fluxo de auth não passa pelo gateway (vai direto ao Entra endpoint) |
| Claim `oid` ausente ou nulo no `X-Entra-OID` | Token obtido via fluxo de conta pessoal (MSA) em vez de conta organizacional; ou `context.User.FindFirst("oid")` com claim name incorreto | Confirmar que App Registration é do tipo "Accounts in this organizational directory only"; usar `context.User.FindFirst("http://schemas.microsoft.com/identity/claims/objectidentifier")` como fallback |
| App Registration sem permissão/scope (AADSTS65001) | Scope `purchase.write` não foi exposto ou não foi consentido | Portal → App Registration → Expose an API → confirmar App ID URI + scope; no MSAL, confirmar que `scopes: ["api://<client-id>/purchase.write"]` |
| Gateway rejeitando JWT válido — chave ou issuer errado (401 com "Signature validation failed") | `opt.Authority` aponta para tenant ID errado ou formato incorreto da URL de discovery | Confirmar `opt.Authority = "https://login.microsoftonline.com/<tenant-id>/v2.0"` — deve incluir `/v2.0`; verificar tenant ID no Portal Entra ID → Overview |
