## Story

**Como** aluno do workshop "Living Lab Azure-Native",
**eu quero** completar as Quartas de Final em sessão única implantando o gateway YARP com todas as policies de borda ativas, configurando a identidade em dois mundos (cliente no Microsoft Entra External ID / CIAM com social login Google + email/OTP; admin no Entra ID workforce com App Roles construído hands-on), e executando a migração prática de usuários v1 → CIAM de forma aditiva — preservando o bcrypt/v1 para comparação,

### Bloco 4 — Migração `users` v1 → CIAM (ADE-007 Inv 6)

15. **AC-15 — Migração v1→CIAM executada como passo hands-on (mecanismo Opção C):** aluno executa a migração das contas de demonstração de `users` (v1/bcrypt) para o tenant CIAM via **self-service sign-up** (mesmo email do `users` v1, usando Google ou email+OTP) + **`UPDATE dbo.users SET entra_oid = @oid WHERE email = @email AND entra_oid IS NULL`** (idempotente por `WHERE entra_oid IS NULL` + índice UNIQUE filtrado `UQ_users_entra_oid`). A migration `phase-04-ciam-link.sql` cria a coluna `users.entra_oid` previamente (pré-aula, instrutor). bcrypt (`users.password`) **permanece intacto** — a senha v1 não é exportável para o CIAM (bcryptjs não aceito pelo External ID); o usuário estabelece credencial nova no CIAM (Google/OTP). Isso é a lição: "no mundo gerenciado, a Microsoft cuida da credencial; você guarda só o `oid`".
    *(Fonte: ADE-007 v1.1 Inv 6; `migration-v1-ciam-design.md` §2 (Opção C escolhida), §3 (bcrypt não exportável), §4 (passo-a-passo); handoff §3 — "passo prático do lab".)*

16. **AC-16 — Coexistência v1/v2 demonstrada pós-migração (query em `users`):** após a migração, o aluno executa a query de verificação em `users` e vê **na mesma linha**: `users.password` (bcrypt v1, intacto), `users.email` e `users.entra_oid` (GUID CIAM recém vinculado). Query esperada (`migration-v1-ciam-design.md` §6):
    ```sql
    SELECT u.id, u.email,
           CASE WHEN u.password LIKE '$2%' THEN 'bcrypt-presente' ELSE 'sem-bcrypt' END AS credencial_v1,
           u.entra_oid AS oid_ciam_v2,
           CASE WHEN u.password IS NOT NULL AND u.entra_oid IS NOT NULL
                THEN 'COEXISTE (v1 bcrypt + v2 CIAM)' ... END AS status_migracao
    FROM dbo.users u WHERE u.email = @email;
    ```
    Resultado esperado: `status_migracao = 'COEXISTE (v1 bcrypt + v2 CIAM)'`. Este é o ápice didático das Quartas — a prova visual na mesma linha de banco de que dois paradigmas de identidade coexistem.
    *(Fonte: ADE-007 v1.1 Inv 3 (eixo cadastro `users.entra_oid`), Inv 4 — coexistência como produto didático; `migration-v1-ciam-design.md` §6 — query de verificação.)*

### auth-flow-validation (o ponto crítico — "o 2º esquema não é só trocar a string")

Confirmei, linha a linha em `Program.cs`, que o desenho dual-issuer é **seguro por construção**:

- **Selector lê issuer NÃO-validado SÓ para rotear, nunca para autorizar.** `AddPolicyScheme("Selector")` usa `ForwardDefaultSelector` que lê `handler.ReadJwtToken(token).Issuer` (sem validar assinatura) **apenas** para escolher entre `Ciam`/`Admin`. A validação real (assinatura RS256/JWKS, `iss`, `aud`, `lifetime`) é 100% do handler concreto escolhido (`AddJwtBearer` Ciam/Admin), com `TokenValidationParameters` explícitos. **Não há nenhuma decisão de segurança baseada no issuer não-validado** — um atacante que forje o `iss` no payload só consegue ser *roteado* ao handler errado, que então **rejeita** o token (assinatura/iss não casam). Correto.
- **Fail-closed em AMBOS os mundos.** Ciam e Admin: tenant+client OBRIGATÓRIOS (`InvalidOperationException` no startup se ausentes), bloqueio explícito de `"common"`, `ValidIssuer` e `ValidAudiences` explícitos (não inferidos só do Authority), `ClockSkew = TimeSpan.Zero`. Carry-forward M-1 do gate S2.2 honrado nos dois esquemas.
- **`AdminOnly` = `RequireAuthenticatedUser().RequireRole("Admin")` SEM fixar esquema.** Decisão deliberada e correta: deixa o selector autenticar, então um **cliente CIAM válido** na rota admin recebe **403** (autenticado, sem a role) — não 401 — preservando a distinção autenticado-mas-não-autorizado. Admin sem role → 403; sem token/ilegível → cai no default Ciam → **401**. Os 4 testes de `DualIssuerTests` + `AdminRejectionTests` provam exatamente esses 4 caminhos (200 admin-com-role / 403 cliente-CIAM / 403 admin-sem-role / 401 sem-token / 401 admin-expirado). Cobertura precisa.
- **`RoleClaimType="roles"` + `MapInboundClaims=false`** no handler Admin: mapeia a App Role do Entra (emitida na claim `roles`) para o role claim type, fazendo `RequireRole("Admin")` casar. Espelhado no fixture de teste. Correto.
- **Anti-spoofing `X-Entra-OID` preservado e issuer-agnóstico.** O transform `Remove(EntraOidHeader)` **sempre** descarta o header de entrada antes de injetar o `oid` extraído do `HttpContext.User` JÁ validado — independente do esquema que autenticou. `SpoofedXEntraOidHeader_FromClient_IsStripped` prova. Inv 2 (pipeline `oid → X-Entra-OID → entra_oid` intocado) confirmada — o gateway é o único guardião do JWT; a Function nunca valida token.
- **Nenhuma rota anônima** exceto `/health` (probe). Toda rota v2 do proxy exige `DefaultPolicy` (autenticado por Ciam OU Admin); token ausente/ilegível → 401.
