## Story

**Como** aluno do workshop "Living Lab Azure-Native" (devs polyglot com background cloud, sem expertise .NET prévia exigida),
**eu quero** completar a Fase 1 do workshop em 6h provisionando Service Bus via Portal Azure e implementando 2 Azure Functions (.NET 8 isolated) que materializam um fluxo de compra v2 desacoplado por mensageria,
**para que** eu compreenda na prática a diferença entre comunicação síncrona vs assíncrona, anatomia do Service Bus, garantias de entrega at-least-once, idempotência no consumer, dead-lettering, e tenha a Function App + queue funcionando como base para todas as fases seguintes.

## Acceptance Criteria

1. **AC-1 — Branch criada:** `phase-01-servicebus-functions` partindo de `main` (estado pré-workshop), com workflow CI/CD próprio `.github/workflows/deploy-phase-01.yml`.

2. **AC-2 — Recursos Azure provisionados via Portal:** aluno demonstra navegação Portal → Resource Group `rg-fifa2026-workshop-<iniciais>` → Service Bus namespace `sb-fifa2026-<iniciais>-<rand>` (Standard tier, East US 2) → queue `tickets-purchase` (lock 30s, max delivery 10, DLQ habilitada).

3. **AC-3 — PurchaseEntryFunction implementada:** Function HTTP trigger `POST /api/v2/purchase` em .NET 8 isolated worker, recebe body `{ matchId, category: VIP|Cat1|Cat2, userId, quantity }`, gera correlationId (GUID), publica mensagem em `tickets-purchase` via output binding declarativo, retorna `{ correlationId, status: "queued" }` em < 100ms.

4. **AC-4 — PurchaseConsumerFunction implementada:** Function Service Bus trigger consome `tickets-purchase`, valida idempotência (`SELECT 1 FROM purchases WHERE correlation_id = @id`), insere registro em SQL Server com `source='v2'` e `correlation_id` preenchido, atualiza status para `completed`.

5. **AC-5 — Schema delta aplicado em pré-workshop:** migration `phase-01.sql` idempotente adiciona colunas `source NVARCHAR(20) NOT NULL DEFAULT 'v1'` + `correlation_id UNIQUEIDENTIFIER NULL` + **UNIQUE constraint filtered `UQ_purchases_correlation_id (correlation_id) WHERE correlation_id IS NOT NULL`** + index. Migração rodada ANTES da aula, não durante. **Conforme ADE-000 Invariante 4 (idempotência robusta via constraint).**

6. **AC-6 — Idempotência verificável via UNIQUE constraint + INSERT-catch (NÃO SELECT-then-INSERT):** enviar a mesma mensagem 2x resulta em **1** registro em SQL. Consumer captura `SqlException ex when ex.Number == 2627` (unique violation) e ignora duplicata silenciosamente. Teste documentado. **Pattern obrigatório por ADE-000 Invariante 4 (evita TOCTOU race condition).**

7. **AC-7 — Dead-letter queue funcional:** mensagem com `matchId` inválido (e.g., 99999) vai para DLQ após 10 tentativas. Aluno demonstra reprocessamento manual via Service Bus Explorer.

8. **AC-8 — Endpoint de status:** `GET /api/v2/purchase/{correlationId}` retorna `{ status: queued|processing|completed|failed, ticketId?: int }` em < 200ms.

9. **AC-9 — Observabilidade:** Application Insights configurado, mostra trace com correlationId atravessando entry + consumer (distributed tracing W3C). Aluno consegue buscar pelo correlationId no App Insights.

10. **AC-10 — CI/CD funcional:** push para `phase-01-servicebus-functions` dispara workflow GitHub Actions que faz `dotnet publish` → deploy via `Azure/functions-action@v1` → smoke test final com `curl` validando `.correlationId` na resposta.

## Dev Notes

### Arquitetura técnica delta (do blueprint)

```
[Browser]
   │ POST /api/v2/purchase
   ▼
[Function App: fifa2026-v2-functions (.NET 8 isolated, Consumption)]
   ├─ PurchaseEntryFunction (HTTP trigger, authLevel: Anonymous em F1)
   │     └─ envia msg → [Service Bus Standard]
   │                       └─ Queue: tickets-purchase
   │                             └─ DLQ auto: tickets-purchase/$DeadLetterQueue
   └─ PurchaseConsumerFunction (Service Bus trigger)
         └─ INSERT em SQL Server (mesma DB do v1)
```

### Schema delta (migration `phase-01.sql`, idempotente — conforme ADE-000 Inv 2 + 4)

```sql
IF NOT EXISTS (SELECT 1 FROM sys.columns WHERE Name='source' AND Object_ID=Object_ID('purchases'))
    ALTER TABLE purchases ADD source NVARCHAR(20) NOT NULL DEFAULT 'v1';

IF NOT EXISTS (SELECT 1 FROM sys.columns WHERE Name='correlation_id' AND Object_ID=Object_ID('purchases'))
    ALTER TABLE purchases ADD correlation_id UNIQUEIDENTIFIER NULL;

-- UNIQUE constraint filtered: idempotência robusta (ADE-000 Inv 4)
IF NOT EXISTS (SELECT 1 FROM sys.indexes WHERE Name='UQ_purchases_correlation_id')
    CREATE UNIQUE INDEX UQ_purchases_correlation_id
        ON purchases(correlation_id)
        WHERE correlation_id IS NOT NULL;
```

> **Por que UNIQUE filtered (não constraint regular?):** `correlation_id` é NULL para rows v1 históricas; UNIQUE regular falharia. Filtered index permite UNIQUE apenas onde NOT NULL.

### Troubleshooting esperado (do blueprint)

| Sintoma | Causa | Mitigação |
|---|---|---|
| Função não consome mensagem | Connection string com `EntityPath=...` no App Setting | Remover `EntityPath` |
| Mensagem reentrega infinita | Lock duration < tempo de processamento | Aumentar para 60s ou `RenewLockAsync` |
| `CommandTimeout` no SQL | Connection pool exaurido | `MaxConcurrentCalls=4` no `host.json` |
| Cold start de 10s na 1ª chamada | Consumption plan free | Aceitar (didático); Premium não nesta fase |
| 401 ao chamar Function | `authLevel: Function` sem chave | Usar `authLevel: Anonymous` em F1 (segurança vira F2 com APIM) |
| `purchase` table not found | Nome real é `purchases` (minúsculo) | Confirmar via `schema.sql` |

### Correlation propagation pattern (preparatório para F6 Flow Visualizer — ADE-000 Inv 5)

- **HTTP hops:** header `traceparent` (W3C Trace Context) + `X-Correlation-ID` (compat)
- **Service Bus hops:** `ApplicationProperties["CorrelationId"]` + .NET `Activity.Current.TraceId` propagação automática (Azure.Messaging.ServiceBus ≥ 7.17)
- **Log hops:** `ILogger.BeginScope(new { CorrelationId = id })` — App Insights captura via `customDimensions.CorrelationId`
- **SQL hops:** coluna `correlation_id UNIQUEIDENTIFIER` (ver schema delta)

Em .NET 8 isolated worker, `System.Diagnostics.Activity.Current` é alimentado automaticamente pelos SDKs (Functions Worker, ServiceBus, HttpClient) — não precisa criar Activity manualmente em F1.

**Decisões e desvios documentados:**
- **[DECISÃO] UNIQUE filtered index (não index regular):** AC-5/Dev Notes/ADE-000 Inv 4 mandam `UNIQUE` index `UQ_purchases_correlation_id` para idempotência. O blueprint F1.④ mostra um index NÃO-unique `IX_purchases_correlation_id` — desatualizado vs. patch arquitetural (Change Log 0.3.0). Segui a story/ADE-000 (source de verdade arquitetural): **UNIQUE filtered index**. Captura também SqlException 2601 (duplicate key em unique index) além de 2627.
- **[DECISÃO] INSERT...SELECT atômico com JOIN:** o INSERT resolve `ticket_category_id`/`unit_price`/`total_price` em uma única statement via `SELECT FROM ticket_categories WHERE match_id=@MatchId AND category=@Category`. Se o par não existe, `rowsAffected==0` → `CategoryNotFound` → consumer re-throw → DLQ (AC-7). Mantém idempotência sem SELECT-then-INSERT.
