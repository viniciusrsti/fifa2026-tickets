## Dev Notes

### Contexto de fluxo — onde o n8n se encaixa

O gateway YARP (F2) injeta `X-Correlation-ID` em cada request (confirmado em `src/Fifa2026.V2.Gateway/Program.cs`, `AddRequestTransform`). O header é propagado pela `PurchaseEntryFunction` para o Service Bus como Application Property. O `PurchaseConsumerFunction` lê a mensagem, grava no SQL e, em F4, passa a disparar o webhook n8n. O n8n é o **quinto hop** do fluxo v2 (Gateway YARP → Function Entry → Service Bus → Function Consumer → n8n → SQL já gravado).

```
[Frontend + MSAL.js (F3)]
   │ Authorization: Bearer <token>
   ▼
[Gateway YARP — Container App por aluno]
   │ injeta X-Correlation-ID (novo GUID se ausente)
   │ valida JWT (AddJwtBearer)
   │ propaga X-Entra-OID
   ▼
[PurchaseEntryFunction — F1]
   │ publica em Service Bus (tickets-purchase)
   │ Application Properties: CorrelationId, EntraOid
   ▼
[Service Bus — tickets-purchase]
   ▼
[PurchaseConsumerFunction — F1]
   │ INSERT idempotente em SQL (entra_oid, correlation_id)
   │ POST fire-and-forget → webhook n8n (F4 NOVO)
   ▼
[n8n — Container App]
   ├─ Switch: VIP vs outros
   ├─ HTTP Request: mock e-mail (httpbin.org)
   └─ Set: log estruturado com correlationId
```

### Container Apps tier para n8n

- Consumption plan (scale-to-zero quando idle — custo ~US$0 em repouso)
- Min replicas: 0, Max replicas: 2
- CPU: 0.5 vCore, Memory: 1Gi
- Ingress: external, HTTPS only, target port 5678

> **Trade-off (honesto):** `latest` = sempre na versão nova, **porém a reprodutibilidade entre aulas não é garantida** (turmas em datas diferentes podem rodar versões diferentes) e o n8n já teve **breaking change em major (2.0)** — https://docs.n8n.io/2-0-breaking-changes/. **Mitigação aprovada:** (a) **validar o workflow `post-purchase-notification` no início de cada aula** antes da demo ao vivo (Task 10.2); (b) cada aula é **gravada com a versão do dia**, então o material reflete o que rodou. Fontes: https://hub.docker.com/r/n8nio/n8n/tags · https://github.com/n8n-io/n8n/releases · https://docs.n8n.io/2-0-breaking-changes/.

### Integração com entraOid (ADE-005)

O `PurchaseConsumerFunction` recebe `EntraOid` como Application Property do Service Bus (propagado pelo gateway via `X-Entra-OID`). Em F4, esse `entraOid` é incluído no payload do webhook ao n8n para personalização condicional futura (e para que o n8n possa identificar o usuário no log). A Function consumer já grava `entra_oid` no SQL (F3).

### Troubleshooting conhecido

| Sintoma | Causa provável | Solução |
|---|---|---|
| n8n não persiste workflows após restart | Azure Files não montado corretamente | Verificar volume mount no CAE; confirmar FQDN da Storage Account |
| Webhook retorna 404 | Workflow não ativado (botão "Active" no n8n UI) | Ativar o workflow e copiar a URL gerada |
| PurchaseConsumerFunction não dispara n8n | `N8N_WEBHOOK_URL` não configurado como App Setting | Adicionar App Setting na Function App via Portal |
| n8n retorna 401 ao smoke test | basic auth não configurado ou env var errada | Verificar env vars N8N_BASIC_AUTH_* no Container App |
| Timeout no webhook (> 5s) | n8n cold start (scale-to-zero) | Fazer warm-up 5min antes da demo; ajustar min replicas para 1 se necessário |

### Testing

**Test approach:**
- Smoke ponta-a-ponta: trigger compra → ver execução no n8n UI dentro de 10s com correlationId correto
- Security: curl sem basic auth → 401; curl com auth correto → 200
- Persistência: redeploy do container n8n → workflows preservados (Azure Files montado)
- Idempotência: mesma compra 2x → n8n chamado apenas 1x (PurchaseConsumer ignora Duplicate)
